## What I cover

Rules for syncing the current project to the canonical Hiroba template skeletons stored in an external Hiroba repo.

## Trigger phrase

Activate this workflow when the user asks to sync the current project from the Hiroba templates or mentions updating instantiated skeletons from the canonical repo.

## Purpose

The current project is an instantiated skeleton (often via symlink). Over time, app-specific content diverges while common infrastructure should stay in sync with the canonical Hiroba template skeletons. This workflow updates common infrastructure safely without overwriting app-owned content.

## Source of truth

The source of truth is an external Hiroba repo. Set a local path to it and compare against the matching skeleton there:

```bash
HIROBA_SOURCE="/path/to/hiroba"
```

- app-template: `$HIROBA_SOURCE/templates/app-template/skeleton/`
- stack-template: `$HIROBA_SOURCE/templates/stack-template/skeleton/`

## Scope

This workflow operates on the current project only. It assumes the project is an instantiated skeleton (or a symlink to one) and is updated in place.

## Git workflow

All changes are made locally in the current repo. Create a branch and commit. If the repo has a remote, open a PR; otherwise keep the commit local.

### 1. Set up the workspace

Run this workflow from the repo you want to update.

### 2. Create a branch from origin/main

Always start from a clean, up-to-date main. Never commit directly to main.

```bash
git branch -D propagate/update-infra 2>/dev/null
git checkout -b propagate/update-infra origin/main
```

### 3. Apply changes

Compare the repo files against the external skeleton and update in place. Use standard file editing. Do not use GitHub API calls.

### 4. Commit

```bash
git add -A
git commit -m "chore(infra): sync common infrastructure with Hiroba template"
```

Prefer scoped conventional commits when possible:

- `ci: update workflow refs to workflows-library@v2`
- `fix(renovate): add missing custom manager for helm values`
- `chore: sync release-please-config with template`
- If multiple scopes are touched: `chore(infra): sync common infrastructure with Hiroba template`

### 5. Push and open a PR (if applicable)

If the repo has a remote, push and open a PR:

```bash
git push -u origin propagate/update-infra
gh pr create --title "chore(infra): sync common infrastructure with Hiroba template" \
  --body "$(cat <<'EOF'
## Summary

Synced infrastructure and tooling updates from the canonical Hiroba template.

### What changed

<!-- List the specific files updated -->

### What was NOT touched

- App-specific content (chart logic, values, Dockerfile, documentation prose, etc.)

### Structural changes (if any)

<!-- List any file moves, renames, new required files, or layout changes -->

> This PR was generated by an automated propagation workflow.
> Review the diff carefully — common infrastructure should match the template.
> App-specific content was preserved; any structural changes are listed above.
EOF
)"
```

### 6. Finish

Do not merge the PR. Leave it for review.

## Identifying the template type

Use these markers to determine app-template vs stack-template:

| Marker | app-template | stack-template |
| --- | --- | --- |
| `helm/base/` directory exists | Yes | No |
| `helm/platform/` directory exists | Yes | No |
| `Dockerfile` exists | Yes (usually) | No |
| `apps/` directory with value overrides | No | Yes |
| `gitops/argocd/root.yaml` (App-of-Apps) | No | Yes |
| `gitops/fluxcd/common/` directory | No | Yes |
| `release-please-config.json` has `helm-base` component | Yes | No |

## Common files: must be kept in sync

These files are infrastructure and must match the template unless there is a documented, intentional reason to diverge. Update them aggressively.

### Artifacthub bootstrap (`.github/workflows/bootstrap-artifacthub.yml`)

- Do not copy the workflow if it does not exist. It is only used for the first commit and removed afterward.

### CI workflows (`.github/workflows/ci.yml`)

- Ensure reusable workflow references point to the correct `7K-Hiroba/workflows-library` ref.
- Ensure job structure matches the template: same jobs, same `if` conditions, same `with` parameters.
- For stack repos, ensure `kubernetes-version` in kubeconform jobs matches the template (or is intentionally pinned with a comment).
- Do not modify app-specific `with` parameters customized at scaffolding time (for example, `image-name`).

### Release workflow (`.github/workflows/release-please.yml`)

- Ensure reusable workflow references match the template.
- Ensure the `permissions` block matches.
- Ensure the `release-please-action` version matches.
- Ensure publish job structure matches (same jobs, same `if` conditions, same `with` parameters).
- Do not modify app-specific `with` parameters (for example, `image-name`, chart names in ArtifactHub jobs).

### Release-please configuration (`release-please-config.json`)

- Sync top-level settings: `separate-pull-requests`, `changelog-type`, `bump-minor-pre-major`, `bump-patch-for-minor-pre-major`, `include-component-in-tag`, `group-pull-requests-by`.
- Ensure component definitions have the correct `release-type`, `include-component-in-tag`, and `changelog-path`.
- Do not add or remove components (component sets are app-specific).
- Do not modify version numbers in `.release-please-manifest.json` (managed by release-please).

### Renovate configuration (`renovate.json5`)

- Ensure `extends` includes `config:recommended` and `:semanticCommits`.
- Ensure `customManagers` match the template (same regex patterns, same datasources).
- Ensure `packageRules` match the template (same grouping, same commit prefixes).
- For app-template repos: ensure the `imageName` regex custom manager is present.
- For stack-template repos: ensure `argocd` and `flux` fileMatch configs are present and kubeconform/pluto custom managers are present.

### Markdownlint (`.markdownlint.yaml`)

- Must match the template exactly (currently `MD013: false`).

### PR template (`.github/PULL_REQUEST_TEMPLATE.md`)

- Must match the corresponding template (app vs stack).

### Issue templates (`.github/ISSUE_TEMPLATE/bug_report.md`)

- Must match the corresponding template.

### AGENT.md

- Sync infrastructure sections: CI/CD workflows, dependency management, markdown linting, commit conventions, technical specifics.
- Do not overwrite the philosophy section if the repo customized it.
- Do not overwrite app-specific architecture or domain sections.
- If the template added new infrastructure sections, add them in the same position.

### LICENSE

- Must match the template.

## App-specific files: protect content, enforce structure

App-specific files are owned by teams. Do not overwrite their content, but do enforce structural changes from the template.

### Structure changes to enforce

- Directory layout changes (move files, do not delete content).
- File naming conventions (rename files, preserve content).
- Required new files (create from template and flag in PR).
- Deprecated common files removed in the template (remove in repo and note in PR).
- Structural updates in config files (update required fields while preserving values).

### Content that must not be overwritten

app-template repos:

- `helm/base/**` (app-specific chart logic)
- `helm/platform/templates/**` (app-specific platform resources)
- `helm/platform/values.yaml`
- `helm/platform/values.schema.json`
- `Dockerfile`
- `compositions/crossplane/**`
- `gitops/**`
- `docs/**` (prose)
- `README.md` (prose)
- `catalog-info.yaml` (metadata values)

stack-template repos:

- `apps/**`
- `gitops/argocd/applications/apps/**`
- `gitops/argocd/applications/common/**`
- `gitops/argocd/root.yaml`
- `gitops/fluxcd/apps/**`
- `gitops/fluxcd/common/**`
- `gitops/fluxcd/git-repository.yaml`
- `docs/**` (prose)
- `README.md` (prose)
- `catalog-info.yaml` (metadata values)

## Handling template variables

Template skeletons contain `${{ values.* }}` and `{% raw %}` blocks. In live repos these were rendered. Preserve substituted values and do not reintroduce raw template syntax.

## Edge cases

### Extra workflows

Leave custom workflows alone; they are app-specific additions.

### Removed components

If a repo removed a component from `release-please-config.json`, do not re-add it.

### Intentional divergence

If a file has a comment like "Intentionally different from template: <reason>", respect it and note the divergence in the PR description.

### Conflicts

If a live repo modified a common file in a conflicting way:

1. Note the conflict in the PR description.
2. Apply template changes where possible.
3. Leave a `TODO` in the code for manual reconciliation if needed.
