# Propagate to All Apps

This file is for AI agents performing bulk updates across repositories that were scaffolded from the Hiroba templates. Activate this workflow when the user says **"propagate to all apps"**.

## Purpose

The Hiroba project scaffolds repositories from two Backstage templates (`app-template` and `stack-template`). After scaffolding, these repos diverge as teams customize them. This workflow propagates **infrastructure and tooling changes** from the canonical templates to all live repos, without disturbing app-specific content.

## Discovering Target Repositories

All Hiroba-scaffolded repos are tagged with GitHub topics for discovery:

- `hiroba-app` — repos created from the app-template
- `hiroba-stack` — repos created from the stack-template

List all target repos:

```bash
gh repo list 7K-Hiroba --topic hiroba-app --json nameWithOwner --jq '.[].nameWithOwner'
gh repo list 7K-Hiroba --topic hiroba-stack --json nameWithOwner --jq '.[].nameWithOwner'
```

If a scaffolded repo is missing its topic, add it:

```bash
gh repo edit 7K-Hiroba/<repo> --add-topic hiroba,hiroba-app   # or hiroba-stack
```

## Git Workflow

All work is done **locally**. The agent clones each repo, makes changes on a branch, and pushes for a PR. Every propagation targets a **single repository** at a time.

### 1. Set up the workspace

Pick a working directory for all clones. If a clone already exists locally, reuse it.

```bash
WORKDIR="$(pwd)"
REPO="7K-Hiroba/example-app"

if [ -d "$WORKDIR/$(basename $REPO)" ]; then
  cd "$WORKDIR/$(basename $REPO)"
  git fetch origin
else
  gh repo clone "$REPO" "$WORKDIR/$(basename $REPO)"
  cd "$WORKDIR/$(basename $REPO)"
fi
```

### 2. Create a branch from origin/main

Always start from a clean, up-to-date main. Never commit directly to main.

```bash
git checkout -b propagate/update-infra origin/main
```

If a `propagate/update-infra` branch already exists (e.g., from a previous run), delete it first:

```bash
git branch -D propagate/update-infra 2>/dev/null
git checkout -b propagate/update-infra origin/main
```

### 3. Apply changes

Read files locally, compare against the canonical template skeleton in the Hiroba repo, and edit them in place. Use standard file editing — no GitHub API calls. See the sections below for what to change and what to preserve.

### 4. Commit

```bash
git add -A
git commit -m "chore(infra): sync common infrastructure with Hiroba template"
```

Use a conventional commit message scoped to what changed:

- `ci: update workflow refs to workflows-library@v2`
- `fix(renovate): add missing custom manager for helm values`
- `chore: sync release-please-config with template`
- If multiple scopes are touched, use `chore(infra): sync common infrastructure with Hiroba template`

### 5. Push and open a PR

```bash
git push -u origin propagate/update-infra
gh pr create --title "chore(infra): sync common infrastructure with Hiroba template" \
  --body "$(cat <<'EOF'
## Summary

Propagated infrastructure and tooling updates from the canonical Hiroba template.

### What changed

<!-- List the specific files updated -->

### What was NOT touched

- App-specific content (chart logic, values, Dockerfile, documentation prose, etc.)

### Structural changes (if any)

<!-- List any file moves, renames, new required files, or layout changes -->

> This PR was generated by an automated propagation workflow.
> Review the diff carefully — common infrastructure should match the template.
> App-specific content was preserved; any structural changes are listed above.
EOF
)"
```

### 6. Move to the next repo

Do NOT merge the PR — leave it for human review. Return to the workspace root and repeat from step 1 for the next repository.

```bash
cd "$WORKDIR"
```

## Identifying the Template Type

Before making changes, determine whether the target repo is an **app-template** or **stack-template** repo. Check for these markers:

| Marker | app-template | stack-template |
| --- | --- | --- |
| `helm/base/` directory exists | Yes | No |
| `helm/platform/` directory exists | Yes | No |
| `Dockerfile` exists | Yes (usually) | No |
| `apps/` directory with value overrides | No | Yes |
| `gitops/argocd/root.yaml` (App-of-Apps) | No | Yes |
| `gitops/fluxcd/common/` directory | No | Yes |
| `release-please-config.json` has `helm-base` component | Yes | No |

Once identified, use the corresponding template skeleton as the source of truth:

- **app-template**: `templates/app-template/skeleton/` in this repo
- **stack-template**: `templates/stack-template/skeleton/` in this repo

## Common Files: MUST Be Kept in Sync

These files are infrastructure. They must match the template unless there is a documented, intentional reason for the target repo to diverge. **Update them aggressively.**

### CI workflows (`.github/workflows/ci.yml`)

- Ensure reusable workflow references point to the correct `7K-Hiroba/workflows-library` ref (e.g., `@v1`, `@main`).
- Ensure job structure matches the template: same jobs, same `if` conditions, same `with` parameters.
- For stack repos: ensure `kubernetes-version` in kubeconform jobs matches the template (or is intentionally pinned to a different cluster version — check for a comment explaining why).
- Do NOT modify app-specific `with` parameters that were customized at scaffolding time (e.g., `image-name` in app-template repos uses the app's actual name, not the template variable `${{ values.name }}`).

### Release workflow (`.github/workflows/release-please.yml`)

- Ensure reusable workflow references match the template.
- Ensure the `permissions` block matches.
- Ensure the `release-please-action` version matches.
- Ensure publish job structure matches (same jobs, same `if` conditions, same `with` parameters).
- Do NOT modify app-specific `with` parameters (e.g., `image-name`, `chart` names in ArtifactHub jobs).

### Release-please configuration (`release-please-config.json`)

- Ensure the **schema**, **top-level settings** match: `separate-pull-requests`, `changelog-type`, `bump-minor-pre-major`, `bump-patch-for-minor-pre-major`, `include-component-in-tag`, `group-pull-requests-by`.
- Ensure the **component definitions** have the correct `release-type`, `include-component-in-tag`, and `changelog-path` for each component.
- Do NOT add or remove components. The set of components is app-specific (an app might not have a Dockerfile or Crossplane compositions).
- Do NOT modify version numbers in `.release-please-manifest.json` — that file is managed by release-please.

### Renovate configuration (`renovate.json5`)

- Ensure `extends` includes `config:recommended` and `:semanticCommits`.
- Ensure `customManagers` match the template (same regex patterns, same datasources).
- Ensure `packageRules` match the template (same grouping, same commit prefixes).
- For app-template repos: ensure the `imageName` regex custom manager is present.
- For stack-template repos: ensure `argocd` and `flux` fileMatch configs are present, and the kubeconform/pluto custom managers are present.

### Markdownlint (`.markdownlint.yaml`)

- Must match the template exactly. Currently: `MD013: false`.

### PR template (`.github/PULL_REQUEST_TEMPLATE.md`)

- Ensure it matches the corresponding template (app vs stack have different checklists).

### Issue templates (`.github/ISSUE_TEMPLATE/bug_report.md`)

- Ensure it matches the corresponding template.

### AGENT.md

- Sync the **infrastructure sections** of AGENT.md: CI/CD workflows, dependency management, markdown linting, commit conventions, technical specifics (API versions, Helm conventions, values schema conventions, unit test conventions, platform chart conventions).
- Do NOT overwrite the **philosophy section** if the repo has customized it.
- Do NOT overwrite app-specific sections that document the repo's unique architecture or domain.
- If the template added entirely new sections (e.g., a new "Gateway API" section), add them to the target repo's AGENT.md in the same position.

### LICENSE

- Must match the template.

## App-Specific Files: Protect Content, Enforce Structure

App-specific files are owned by the app team. During propagation, **never overwrite the content** teams have written (their prose, their values, their custom templates). However, **structure is not content** — if the template changes how files are organized, named, or which files must exist, enforce that structure across all repos.

### What "structure" means

- **Directory layout**: If the template moves `docs/` from a flat layout to `docs/guides/`, `docs/reference/`, etc., restructure the target repo to match. Move existing files into the correct subdirectories. Do not delete content — relocate it.
- **File naming conventions**: If the template renames `docs/index.md` to `docs/overview.md`, rename it in the target repo too. Preserve the file's content.
- **Required files**: If the template adds a new required file (e.g., `docs/architecture.md` or a new `_helpers.tpl` partial), create it in the target repo. Use the template's version as a starting point, but flag it in the PR description so the team can fill in app-specific details.
- **File removals**: If the template removes a file that was previously common (e.g., a deprecated config file), remove it from the target repo too. Note the removal in the PR description.
- **Structural patterns in config files**: If `catalog-info.yaml` gains new required fields or changes its `spec` structure, update the structure while preserving app-specific values (name, description, owner, etc.).

### What "content" means (do not overwrite)

Content is the app-team's domain-specific work:

**app-template repos:**

| Path | Protected content |
| --- | --- |
| `helm/base/**` | Chart values, templates, schema, tests — all app-specific logic |
| `helm/platform/templates/**` | Platform resource definitions customized per app |
| `helm/platform/values.yaml` | App-specific platform configuration values |
| `helm/platform/values.schema.json` | App-specific schema definitions |
| `Dockerfile` | App-specific build instructions |
| `compositions/crossplane/**` | App-specific Crossplane resources |
| `gitops/**` | App-specific manifest references (app name, chart versions, repo URLs) |
| `docs/**` | Written documentation prose (but structure/layout CAN change) |
| `README.md` | Written prose (but required sections CAN be enforced) |
| `catalog-info.yaml` | Metadata values: name, description, owner (but spec structure CAN change) |

**stack-template repos:**

| Path | Protected content |
| --- | --- |
| `apps/**` | Stack-specific value overrides per app |
| `gitops/argocd/applications/apps/**` | Stack-specific app deployments |
| `gitops/argocd/applications/common/**` | Stack's chosen operators and their config |
| `gitops/argocd/root.yaml` | Stack-specific references |
| `gitops/fluxcd/apps/**` | Stack-specific app Kustomizations |
| `gitops/fluxcd/common/**` | Stack's chosen operators and their config |
| `gitops/fluxcd/git-repository.yaml` | Stack-specific repo URL |
| `docs/**` | Written documentation prose (but structure/layout CAN change) |
| `README.md` | Written prose (but required sections CAN be enforced) |
| `catalog-info.yaml` | Metadata values: name, description, owner (but spec structure CAN change) |

### Structural change examples

**Example 1 — Docs restructure:**
Template changes from flat `docs/` to `docs/guides/` + `docs/reference/`. Target repo has `docs/deployment.md` and `docs/troubleshooting.md`. Agent moves `deployment.md` to `docs/guides/deployment.md` and `troubleshooting.md` to `docs/reference/troubleshooting.md`. Content is preserved, layout matches the template.

**Example 2 — New required file:**
Template adds `docs/adr/` for architecture decision records with a `docs/adr/README.md` template. Agent creates the directory and the README in the target repo. Notes in the PR: "New `docs/adr/` directory added per template — team should add ADRs as needed."

**Example 3 — catalog-info.yaml spec change:**
Template adds a new `spec.lifecycle` field. Agent adds `spec.lifecycle: production` (or whatever the template default is) to the target repo's `catalog-info.yaml`, preserving all existing fields. Notes in the PR: "Added `spec.lifecycle` field — review and adjust if this component is not in production."

## Handling Template Variables

The template skeletons use Jinja2/Nunjucks variables like `${{ values.name }}` and `{% raw %}` / `{% endraw %}` blocks. When comparing a template file against a live repo:

- Template variables will have been substituted with actual values at scaffolding time. For example, `${{ values.name }}` becomes `radarr`.
- `{% raw %}` and `{% endraw %}` tags will have been stripped — the content between them is literal in the live repo.
- When propagating, preserve the live repo's substituted values. Only update the surrounding structure.

## Edge Cases

### Repo has extra workflows

Some repos may have added custom workflows beyond `ci.yml` and `release-please.yml` (e.g., a nightly build, a custom deploy workflow). Leave these alone — they are app-specific additions.

### Repo has removed template components

If a repo removed a component from `release-please-config.json` (e.g., no Crossplane), do not re-add it. The component set is an app-level decision.

### Repo has intentional config divergence

If a file has a comment like `# Intentionally different from template: <reason>`, respect it. Note the divergence in the PR description so reviewers are aware.

### Conflicting changes

If the live repo has modified a common file in a way that conflicts with the template update, do NOT force-overwrite. Instead:

1. Note the conflict in the PR description.
2. Apply the template change where possible.
3. Leave a `TODO` comment in the code if a section needs manual reconciliation.
